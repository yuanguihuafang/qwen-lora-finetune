01 · PHASE

前期准备 — 环境 & 选型

核心问题
在动手之前，必须回答三个问题：微调还是 RAG？用哪个基座模型？在哪里训练？ 这三个决策直接决定后续所有资源投入。

是否真的需要微调？

场景推荐方案原因
需要注入外部最新知识RAG微调无法更新参数知识，成本高
需要特定风格 / 格式输出✅ 微调风格一致性靠参数记忆最稳定
领域专业词汇 / 推理能力✅ 微调让模型真正"学会"而不是检索
减少 System Prompt 长度✅ 微调将规则烧入权重，降低推理成本
多步骤工具调用流程✅ 微调 + RAG混合架构效果最佳

02 · PHASE

数据工程 — 质量决定上限

⚠️ 最常见的失败原因
80% 的微调失败来自数据质量问题，而非模型或超参数。宁可花更多时间在数据清洗上，也不要用脏数据反复训练。
1
确定数据格式（Chat / Instruction / Completion）
微调格式需与基座模型的 chat template 一致。Llama 系列使用 <|begin_of_text|> 格式；Qwen 使用 <|im_start|>。格式错误会导致模型学到无效模式。
chat-templatetokenization
2
数据收集 — 多源整合
优先使用真实用户数据；其次使用强模型（GPT-4o / Claude）生成高质量合成数据；对于中文任务可使用 BELLE、Chinese-LLaMA 等开源数据集作为起点。
synthetic-dataaugmentationcrawl
3
数据清洗 Pipeline
去重（MinHash/SimHash）→ 过滤低质量（长度、困惑度）→ PII 脱敏（姓名/手机/身份证）→ 格式规范化 → 人工抽检 2–5%。
deduppii-removalquality-filter
4
数据集拆分与统计
推荐比例：Train 90% / Val 8% / Test 2%。统计 token 分布、平均长度，确保无严重偏斜。验证集要覆盖所有子任务类型。
train-val-teststatistics

03 · PHASE

微调训练 — LoRA / QLoRA 实战

LoRA
Low-Rank Adaptation：仅训练低秩分解矩阵 A、B，参数量约为全量的 0.1–1%。适合显存充足（≥24GB）的 FP16 训练。
QLoRA
Quantized LoRA：基座模型 4-bit 量化（NF4）加载，LoRA 适配器 BF16 训练。单张 24GB 卡可微调 70B 模型。
QLoRA 完整训练脚本 · train_qlora.py
from transformers import AutoModelForCausalLM, AutoTokenizer, BitsAndBytesConfig
from peft          import LoraConfig, get_peft_model, TaskType
from trl           import SFTTrainer, SFTConfig
from datasets      import load_dataset
import torch

# ── 1. 量化配置 ──────────────────────────────────
bnb_cfg = BitsAndBytesConfig(
    load_in_4bit              = True,
    bnb_4bit_compute_dtype    = torch.bfloat16,
    bnb_4bit_quant_type       = "nf4",
    bnb_4bit_use_double_quant = True,
)

# ── 2. 加载模型 & Tokenizer ──────────────────────
MODEL_ID = "Qwen/Qwen2.5-7B-Instruct"
model = AutoModelForCausalLM.from_pretrained(
    MODEL_ID, quantization_config=bnb_cfg, device_map="auto"
)
tokenizer = AutoTokenizer.from_pretrained(MODEL_ID)

# ── 3. LoRA 配置 ─────────────────────────────────
lora_cfg = LoraConfig(
    r              = 16,         # 秩，越大表达能力越强但参数更多
    lora_alpha     = 32,         # 缩放因子，通常 = 2×r
    target_modules = ["q_proj", "k_proj", "v_proj", "o_proj",
                      "gate_proj", "up_proj", "down_proj"],
    lora_dropout   = 0.05,
    task_type      = TaskType.CAUSAL_LM,
)
model = get_peft_model(model, lora_cfg)
model.print_trainable_parameters()  # 确认可训练参数比例

# ── 4. 训练配置 ──────────────────────────────────
training_args = SFTConfig(
    output_dir              = "./output",
    num_train_epochs        = 3,
    per_device_train_batch_size = 2,
    gradient_accumulation_steps = 8,   # 等效 batch_size=16
    learning_rate           = 2e-4,
    lr_scheduler_type       = "cosine",
    warmup_ratio            = 0.05,
    bf16                    = True,
    logging_steps           = 10,
    save_strategy           = "epoch",
    eval_strategy           = "epoch",
    report_to               = "wandb",
    max_seq_length          = 2048,
)

# ── 5. 启动训练 ──────────────────────────────────
dataset = load_dataset("json", data_files={"train": "train.jsonl", "validation": "val.jsonl"})
trainer = SFTTrainer(model=model, args=training_args,
                     train_dataset=dataset["train"],
                     eval_dataset=dataset["validation"])
trainer.train()

# ── 6. 合并保存 ──────────────────────────────────
merged = trainer.model.merge_and_unload()
merged.save_pretrained("./merged_model")
tokenizer.save_pretrained("./merged_model")

超参数调优指南

参数推荐范围调优方向
learning_rate1e-4 ~ 3e-4loss 不下降 → 调大；震荡 → 调小
LoRA rank (r)8 / 16 / 32任务越复杂 → 越大；显存不足 → 调小
lora_alpha通常 = 2×r保持 alpha/r 比值稳定即可
batch size (等效)16 ~ 128小 batch 收敛慢但泛化好
num_epochs1 ~ 5val loss 上升即停止（早停）
max_seq_length1024 ~ 4096超出上限的样本会被截断
04 · PHASE

评估验证 — 多维度质量检验

💡 评估黄金法则
自动指标（ROUGE/BLEU）只是参考，人工评估 + 模型评估（LLM-as-Judge）才是最终标准。始终在保留的 holdout 测试集上评估，绝不用训练集中的数据。
A
自动指标评估
ROUGE-L（摘要/文本生成）、BLEU（翻译）、F1（信息抽取）、Perplexity（语言建模）。使用 HuggingFace evaluate 库快速计算，与 baseline 模型对比。
rougebleuperplexity
B
LLM-as-Judge 评估
用 GPT-4o 或 Claude 作为裁判，对微调模型和原始模型输出进行 A/B 对比评分（1-10 分），覆盖：准确性、流畅性、遵循指令程度。成本低、可扩展。
llm-judgepairwisepreference
C
灾难性遗忘检测
微调后必须用通用基准（MMLU / C-Eval）检测模型是否严重退化。如果通用能力下降 >5%，需要减小 LoRA rank 或加入通用数据混合训练。
mmluc-evalcatastrophic-forgetting
D
安全性 & 边界测试
构造对抗性输入（越狱尝试、歧义指令、极端边界），确保微调未引入安全漏洞。建立 red-teaming 测试集，每次更新模型必须通过。
red-teamsafetyrobustness
LLM-as-Judge 评估 · eval_judge.py
import anthropic, json

client = anthropic.Anthropic()

JUDGE_PROMPT = """
你是一位严格的评估专家。请对以下两个回答进行对比评分（1-10分）。

问题：{question}
回答A（原始模型）：{answer_a}
回答B（微调模型）：{answer_b}

评估维度：准确性、专业性、格式规范性。
请只输出 JSON：{{"winner": "A"或"B"或"tie", "score_a": 分数, "score_b": 分数, "reason": "简要原因"}}
"""

def judge(question, answer_a, answer_b):
    msg = client.messages.create(
        model="claude-opus-4-5",
        max_tokens=512,
        messages=[{"role": "user",
                   "content": JUDGE_PROMPT.format(
                       question=question, answer_a=answer_a, answer_b=answer_b)}]
    )
    return json.loads(msg.content[0].text)

from openai import OpenAI

client = OpenAI(base_url="http://localhost:8000/v1", api_key="dummy")

response = client.chat.completions.create(
    model="my-model",
    messages=[
        {"role": "system", "content": "你是专业的法律助理。"},

Prometheus 监控指标 · metrics.py
from prometheus_client import Counter, Histogram, Gauge

# 核心监控指标
REQUEST_COUNT   = Counter("llm_requests_total", "Total API requests", ["status"])
TOKEN_COUNT     = Counter("llm_tokens_total",   "Total tokens",      ["type"])
LATENCY_TTFT    = Histogram("llm_ttft_seconds",  "Time to first token")
LATENCY_TOTAL   = Histogram("llm_e2e_seconds",   "End-to-end latency")
GPU_UTIL        = Gauge("llm_gpu_utilization", "GPU utilization %")

# 在每次请求中埋点
def track_request(func):
    async def wrapper(*args, **kwargs):
        with LATENCY_TOTAL.time():
            result = await func(*args, **kwargs)
        REQUEST_COUNT.labels(status="success").inc()
        return result
    return wrapper
